4.11.4 Transport

All exchanges can be transported over these protocols:

  1. As a physical file with manual exchange via eMail, Harddrives, CDE's or similar
  2. As a data structure in the payload of an online exchange, eg. an API, a BCF message, a messaging system or similar.

This standard does not define how exactly the exchange is done. It defines the standard for the structure of the file or the payload. The file or the payload can be stored in a compressd format according to the specification. It is allowed to embed an IFC file in Step format (.ifc) or in XML format Format (.ifcXML) in an ifcZIP file.The specification contains the following principles:

  1. The ifcZIP file contains exactly one model file (.ifc or .ifcxml). The file extension is ifcZIP
  2. According to this rule, it is allowed to embed additional other files in an ifcZIP file. Therefore, an ifcZIP file can be a container, which can transport many additional files in itself. This container can transport related photo documentations, plant documentation, permits and notifications, operating documents, maintenance reports and other model related documents.

The specification of the ifcZIP format allows one model file in the root folder of the ifcZIP file. The documents can be stored relative to the root folder and references with the entity IfcDocumentInformation to the model. For each file transported in the container, an element of type IfcDocumentInformation is written.
